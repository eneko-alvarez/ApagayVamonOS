Buscador vulnerable — XSS Reflejado

El valor del campo de búsqueda se inserta directamente en el DOM con innerHTML sin ningún tipo de sanitización.

📖 ¿Cómo funciona?

El script toma el valor del input y lo escribe directamente como HTML en el DOM: 

/* ── VECTOR 1: XSS Reflejado ── */
function doSearch() {
  document.getElementById('search-output').innerHTML
    = document.getElementById('search-input').value;
}

Payload de ejemplo:

<img src=x onerror="alert('XSS!')">
  • El navegador parsea el HTML inyectado
  • La imagen falla → dispara onerror
  • El alert() se ejecuta en el contexto de la página

Defensa

✅ Solución — Cómo arreglarlo

Dos enfoques complementarios: escapar la salida y usar APIs seguras del DOM.

🛡️ Medidas de mitigación

Fix 1 — Reemplazar innerHTML por textContent

❌ Vulnerable

function doSearch() {
  output.innerHTML
    = input.value; // 💥 XSS
}

✅ Seguro

function doSearch() {
  output.textContent
    = input.value; // ✅ seguro
}

Fix 2 — Función escapeHTML()

function escapeHTML(s) {
  return s.replace(/&/g,  '&amp;')
           .replace(/</g,  '&lt;')
           .replace(/>/g,  '&gt;')
           .replace(/"/g,  '&quot;')
           .replace(/'/g,  '&#039;');
}
function doSearch() {
  output.innerHTML = escapeHTML(input.value);
}

Fix 3 — Content Security Policy (CSP)

# Cabecera HTTP (Apache / Nginx / Express)
Content-Security-Policy:
  default-src 'self';
  script-src  'self';        /* sin 'unsafe-inline' */
  img-src     'self' https:; /* solo imágenes HTTPS  */
  object-src  'none';
  • Sin 'unsafe-inline': scripts inline inyectados no se ejecutan
  • img-src 'self' https:: bloquea src=x con esquema inválido
  • object-src 'none': elimina el vector de objetos Flash/plugin
